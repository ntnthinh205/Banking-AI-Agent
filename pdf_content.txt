1 Project requirements
Figure 1: High-level workflow of AI Agent in this project
The objective of this lab is to design and implement a simpleAI agentic workflowfor
customer support in the banking domain. The system should be able to receive a customer
message, identify the corresponding intent, retrieve relevant policy information, generate a draft
response, and determine whether the case can be handled automatically or should be escalated
to human staff. Specifically, students are required to complete the following tasks:
•Build an AI agentic pipeline for handling customer issues in the banking domain.
•Implement at least oneintent detection nodeas a core component of the workflow.
•Add several additional basic nodes to complete the workflow, such as priority estimation,
policy retrieval, response drafting, validation, or escalation.
•Integrate an LLM-based component if needed. If students choose to use an AI model, they
are encouraged to usegpt-oss-20b, an open-weight model designed for local, low-latency,
and specialized use cases. OpenAI states thatgpt-oss-20bis suitable for local inference
and agentic workflows, but it is not served through the OpenAI API.
•Demonstrate the complete workflow on a set of example banking customer requests.

2 Task requirements
2.1 System design
•Students must design an agentic workflow consisting of multiple nodes. The system does
not need to be a fully autonomous agent; a structured workflow or a simple state-machine
design is sufficient.
•The workflow must include the following mandatory node:
– Intent Detection Node: identifies the customer intent from the input message.
– Priority or Risk Detection Node: determines whether the issue is low, medium,
or high priority.
– Policy Retrieval Node: retrieves a relevant F AQ entry, policy snippet, or support
guideline.
– Response Drafting Node: generates a draft reply for the customer.
– V alidation Node: checks whether the generated response is consistent with the policy
and whether important information is missing.
– Escalation Node: decides whether the case should be transferred to a human support
agent.
2.2 Run LLM model
•Student can make use of GPUs free of Google Colab to rungpt-oss-20bby using Ollama.
Afterward, use Pinggy to forward the local port of Ollama (localhost:11434) to a public
port (typical type: http://*.a.free.pinggy.link) that students can call as an API of LLM.
Refer to the given notebook:Ollama-Pinggy.ipynb.
•Students are encouraged to read Ollama documents for more details.

•Provide the entry point for running the application.

•Define the input and output schemas used throughout the system.

app/data/policies.py
•Store the policy data or F AQ snippets used by the system.

just need to create some dummy data that simulates real data in this project rather than
building a database.

app/clients/base.py
•Define the base client interface for model calling.
•Since this project only uses Ollama, this file can still be used as an abstraction layer so
that all model-calling components follow a consistent interface.
app/clients/ollama_client.py
•Implement the Ollama client used to call the language model.

app/nodes/intent_node.py
•Implement the intent detection node.
•This node must use the fine-tuned model obtained in Lab 2.
•Its responsibility is to receive a customer message and return the predicted banking intent.
•Students may design this node as a wrapper around the Lab 2 inference class or checkpoint
loader.
•This is the core node of the workflow and should be treated as the main specialized classifier
in the system.
app/nodes/priority_node.py
•Implement a simple priority or risk detection node.
•This node may use rules, keywords, or lightweight logic to classify whether a case is low,
medium, or high priority.
•Example: issues involving suspicious transactions, blocked accounts, or lost money may be
marked as high priority.
app/nodes/policy_node.py
•Implement the policy retrieval node.
•Given the predicted intent, this node returns the relevant policy or guideline frompolicies.py.
•The purpose of this node is to ground the final response in predefined support knowledge.

app/nodes/draft_node.py
•Implement the response drafting node.
•This node may call Ollama to generate a draft reply to the customer.
•Students are encouraged to usegpt-ossin Ollama for this component.
•The draft should take into account the customer message, the detected intent, the priority
level, and the retrieved policy.
•The output may include the draft reply, missing information, and the next suggested action.
app/nodes/validation_node.py
•Implement a validation node for checking whether the generated draft is acceptable.
•This node may verify whether the response is too short, whether important information is
missing, or whether the detected intent confidence is too low.
•Students may keep this node simple, but it should help improve reliability and transparency.
app/nodes/router_node.py
•Implement the final decision node of the workflow.
•This node decides whether the system should:
–Send a reply directly.
–Ask the customer for more information.
–Or escalate the case to a human support team.
•The routing decision should be based on the outputs of the previous nodes.
app/agent/orchestrator.py
•Implement the main workflow controller.
•This file is responsible for calling all nodes in the correct order:
–Intent detection.
–Priority detection.
